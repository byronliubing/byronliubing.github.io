ARMv8-A架构

发表于 2025-05-02 更新于 2025-05-09 本文字数： 5.2k 阅读时长 ≈ 19 分钟

此篇将记录一些关于ARMv8-A架构相关的知识及使用说明。

一、启动流程

在支持TrustZone时，需要ATF的支持，ATF的流程是启动安全世界的，用来保证后续启动流程的鉴权是安全的。其启动流程大致为分为：

其中BL1为BOOTROM程序固化在芯片内部，芯片出产后就不能再修改，所以一般不涉及常规外设初始化等易变动的事情。但是会根据芯片所支持的Flash列表来集成对应的Flash驱动，这样可以从Flash中加载后续的固件到SRAM中运行，即下一个固件为BL2，也包括固件验签、安全启动等流程。

BL2运行在SecureWorld的El1（记作S-EL1），其主要初始化DDR等设备，然后加载后续的BL31 BL32 BL33到DDR中运行。由于BL2运行在S-EL1，所以BL1中还需要完成“SMC”指令的vector实现，否则无法返回。

BL31是runtime services，需要在内存中驻留的，而BL1 BL2用完就可以释放了，运行在EL3中。该层级中提供需要处理很多事情，包括页表创建、cache使能、对应vector_el3的配置级实现、EL3的中断路由和上下文切换等。

BL32是TRUST-OS，运行在S-EL1，提供安全世界的操作系统，用于完成配置的安全业务

BL33则是我们熟知的UBOOT，然后UBOOT加载NS-EL1的OS继续运行。

BL2与UBOOT的SPL关系

spl被称为secondary program loader，在启动链中一般由bootrom加载而作为第二级启动镜像（bl2），它主要用于完成一些基础模块和ddr的初始化，以及加载下一级镜像uboot。由于spl需要被加载到sram中执行，对于有些sram size比较小的系统，可能无法放入整个spl镜像，tpl即是为了解决该问题引入的。加入了tpl之后，可将spl的功能进一步划分为两部分，如spl包含ddr初始化相关代码，而tpl包含镜像加载相关驱动，从而减少spl镜像的size。此时启动流程可被设计为如下方式： bootrom –> spl（init ddr） –> bootrom –> tpl（load and run uboot）–> uboot

当然很多时候tpl是可以省略的，即模型变为：

bootrom –> spl（init ddr） –> uboot —> Linux

应对某些需要快速启动的场景，模型甚至可以升级为：（uboot也不需要，由Linux接管驱动初始化等）

bootrom –> spl（init ddr） –> Linux

如果系统需要支持secure核non secure两种安全状态的话，则必须先从secure空间开始启动，此时ATF貌似就变得必须了。

冷启动时

处理器上电复位后，运行在何种执行状态是如何决定的呢？是由处理器内部硬件AA64nAA32决定的，这个信号是芯片设计时进行配置的。在处理器上电复位时，通过采样 AA64nAA32的信号值决定进入对应的执行状态。我们在ARMv8-A架构处理器的TRM手册中可以具体了解AA64nAA32信号。（A.4 Configuration signals章节）

  • 对于AArch32执行状态，入口地址由寄存器SCTLR.V值决定，而SCTLR.V值由处理器内部硬件信号VINITHI 决定的，这个信号是芯片设计时进行配置的。

  • 对于AArch64执行状态，入口地址由处理器内部硬件信号RVBARADDR决定，这个信号是芯片设计时进行配置的。在处理器上电复位时，通过采样RVBARADDR的信号值，读入到RVBAR_EL3寄存器，获取入口地址。一般会把内部BOOTROM固化到RVBARADDR处，这样上电之后就直接跳转到BOOTROM中运行了。

三、中断处理

GICv3整体框架：

中断号定义

  • 本来1024–8191为保留中断号，在该基础上，引出两个扩展PPI（64个）与扩展SPI（1024个），但对一般SOC而言，用不了这么多

  • 上述描述比较清楚了，当GIC发送的fiq中断路由时，el3需要做世界切换，此时获取到的INTID值为上述的。然后运行在EL3的BL31中的对应中断处理函数，需要判断该INTID并做对应的切换操作

  • 关于NMI：

    • 目前只有GICv3.3和GICv4.2中添加了支持—NMIs are only presented as having superpriority when being presented to their owning Security state and Exception level.
    • Group 0 interrupts cannot be configured as NMIs.（到EL3的中断）

两种中断模式

目前PE中支持两种中断模式：IRQ和FIQ。在ARMv8之前，通常表示两种不同优先级的中断：即FIQ表示快速中断，优先级比IRQ较高。但在ARMv8中，任意时刻只会有一个被触发，要么IRQ要么FIQ，通常用来表示中断路由方式。

  • 外设直接set or clear Controller的寄存器。NOTE: LPIs are only supported when GICD_CTLR.ARE_NS==1

中断触发的两种方式：边沿触发与电平触发

中断group：

  • GICD_CTLR.DS == 0：Two Security states (Secure and Non-secure) are supported

  • GICD_CTLR.DS == 1：Only a single Security state is supported

    （Disable Security. 只有在secure 状态下可操作该bit）

    上述表比较重要，承载着中断路由的配置方式：

    当一个non secure el1的中断触发时（配置的为Non_Secure Group 1）：

    1、当前CPU处于Secure EL1，根据上表，触发FIQ信号。需要先路由到EL3，然后转给NS EL1处理

    2、当前CPU处于NS EL1时，触发IRQ信号，不用额外路由，直接处理即可

中断路由：

安全世界与非安全世界的寄存器很多都是bank的，例如VBAR_EL1和VBAR_EL2，所以不存在冲突的情况。

关于中断路由的问题，可以根据GIC的配置对PE发出IRQ或者FIQ中断，一般FIQ中断都是路由在EL3的，这是怎么处理的呢？

1、从EL3返回时，会对SCR.NS位进行操作，选择回到S或NS世界

2、在处理FIQ也一样，GIC发送FIQ至EL3时，根据BL31设置的vector_el3跳转到对应的FIQ处理函数中，此时读到的GIC INTID是一个特殊值（1020，1021等），根据该值需要跳转到对应的handler。每类INTR_TYPE注册时都会计算SCR值，更新NS位。

3、然后返回到注册的handler中，此时需要注意，在进入EL3的VECTOR时已经关闭了DAIF，但在EL3返回到lower EL时，会根据spsr_el3值可能重新打开中断，但是eret返回时的PC值已经指定。（其实就是模拟了一次eret，至于异常进入时的pc（elr_el3）和cpsr（spsr_el3）都是在注册时handler中指定的或实时读取的）

中断完成模式

为了提高中断响应的实时性，GIC将中断完成分解为两个阶段：

  • 优先级重置（Priority drop）：将当前中断屏蔽的优先级进行重置，从而能够响应更低优先级的中断。
  • 中断无效（Deactivation）：将当前对应的中断状态置为Inactivate，从而可以重新响应该优先级的中断。

中断嵌套说明

​ 目前GICv3实现中断嵌套需要两个条件：

  • interrupt的priority必须高于 Priority Mask register (ICC_PMR_EL1) ，只有高于才能pending。

    Only interrupts with a higher priority than the value in this register are signaled to the PE.

  • pending interrupt的group priority必须高于Running Priority（ICC_RPR_EL1）

​ 当PE响应acknowledges一个中断时，running priority会储存当前中断priority，running priority要么是当前active interrupt的group priority，要么是idle priority (0xFF）

GICv3初始化时，将priority分成两部分：支持16层中断嵌套(可调，最高可以做到7bits，同时需要考虑中断嵌套层级以及栈大小）

四、关于异常的说明

  • 异常时，会将PC及PSTATE保存到对应的EL_ELn及SPSR_ELn中，用于异常返回时恢复

  • The Fault Address Register (FAR_ELn) holds the faulting virtual address for all synchronous instruction and Data Aborts and alignment faults

  • The Exception Syndrome Register (ESR_ELn) gives information about the reasons for the exception

  • 针对实现了EL2/EL3的系统，中断是可以路由到更高的异常等级中处理，然后由高异常等级中的处理来分配。这也是实现Hypervisor的一种常用手段。将一些需要处理的中断发送到EL2，EL2初处理然后再分发到其他虚拟机中。各个虚拟机中可以直接使用vIRQ来实现

异常返回

ERET指令用于从异常处理流程中返回，在介绍异常返回流程之前，我们先看一下armv8异常跳转的流程。Armv8触发异常或中断后硬件将会执行以下操作：

（1）将PSTATE寄存器的内容保存到SPSR_ELx中，其中x表示异常进入的异常等级，如smc异常将会陷入EL3，因此相应的寄存器就为SPSR_EL3

（2）将异常处理完成后需要返回的地址保存在ELR_ELx中

（3）设置中断和异常掩码DAIF，以关闭所有中断和异常

（4）若异常是同步异常或SError中断，异常状态信息将被保存在ESR_ELx中，该信息可用于分析异常发生的原因

（5）若异常为指令异常、数据异常或对齐错误等，则触发异常对应的内存地址将被保存到FAR_ELx寄存器中

（6）栈指针切换为目标异常等级的栈指针寄存器SP_ELx（此处不管ELx时设置的spsel值是多少）

（7）程序跳转到对应的异常处理入口，执行异常处理程序 在异常执行完成后，可通过ERET指令返回被异常中断程序的断点处继续执行，该指令将使硬件执行以下操作：

（1）用SPSR_ELx寄存器的内容恢复PSTATE寄存器

（2）用ELR_ELx寄存器的内容恢复PC值

从以上流程可以看到，执行ERET指令后程序的执行流将由SPSR_ELx和ELR_ELx决定。因此，我们在执行镜像之间的跳转，只要在ERET之前将ELR_ELx设置为待跳转镜像的入口地址，并设置正确的SPSR_ELx即可完成。

五、系统时钟

每个Arm核都有4个私有定时器，每个定时器都有一个比较寄存器、一个定时寄存器、一个控制寄存器，所以一共应该有12个寄存器可以操作，将它们的命名总结如下：

这7个timer 的”值”都来源自2个counter

• cntpct_el0，是Physical Count register； cntp，cnthp 基于cntpct_el0

• cntvct_el0，也就是Virtual Count register；cntv，cnthv 基于cntvct_el0

其实cntvct_el0 和cntpct_el0也是有关系的，取决于hypervisor的配置:

• cntvct_el0 = cntpct_el0 - cntvoff_el2

• 或者cntvct_el0 = cntpct_el0

• 当然如果硬件没有实现EL2，cntvct_el0 自然也是等于 cntpct_el0

可以把cntpct_el0/cntvct_el0理解为一个递增的时钟滴答，但他不提供任何定时、中断、enable/disable 等功能，提供这些功能的是上面7个Timer。

每个timer都有一组配套的寄存器：（以cnthp为例）

1、timer value寄存器：cnthp_tval_el2

2、compare value寄存器：cnthp_cval_el2

3、控制寄存器：cnthp_ctl_el2

timer触发条件：

物理计数器（对于cnthp是：cntpct_el0) - compare value(对于cnthp是:cnthp_cval_el2) >= 0

七、特殊指令说明

关于跳转指令

ldr指令与ldr伪指令（带=号的）：

pre-index/post-index。

pre-index是先修改基址寄存器，然后再访问内存。

post-index则是先访问内存，再修改基址寄存器。后变基经常用来出栈。该指令用于从当前SP位置开始加载值到寄存器，然后将SP进行偏移。

• stp x1, x0, [sp, #-16]!

1、sp’ = sp - 16

2、将 x1 填入 sp’ 位置，将 x0 填入 sp’ + 8 位置

• ldp x29, x28, [sp], #16

1、将 sp 位置处读取8字节到x29，将 sp + 8位置处读取8字节到x28

2、sp’ = sp + 16

上述两个示例，stp/ldp能同时操作两个寄存器，应该是不存在中间步骤（sp+8 或者 sp-8

CurrentEL说明

PSTATE.EL字段保存当前PE的异常等级，但复位时该字段表示系统所支持的最高的异常等级，大致需要从最高异常等级开始初始化。

而CurrentEL表示当前的异常等级，其逻辑也是获取PSTATE.EL字段。即复位后表示当前系统能支持的最高异常等级，CurrentEL表示当前的异常等级。可以读该寄存器来获取当前所处等级。

VHE说明

在HCR.E2H配置下，可以使能VHE功能，但是前提是设备支持VHE（***When FEAT_VHE is implemented***），但是在Linux中的启动流程中：

1、在head.S的init_el2中，先统一关闭VHE功能

2、然后在启动的最后阶段__primary_switched —> finalise_el2 会再判断是否进入VHE，即EL2启动
